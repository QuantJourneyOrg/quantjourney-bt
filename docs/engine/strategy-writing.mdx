# Writing Strategies

This page is the practical entry point for writing a QuantJourney strategy.
Start here when you have an investment idea and need to turn it into a runnable
backtest.

QuantJourney has two main strategy shapes:

- Weight mode: the strategy returns target portfolio exposure.
- Order mode: the strategy submits explicit orders and lets the fill engine
  simulate execution.

<div className="qj-callout warning">
  <div className="qj-callout-title">Core rule</div>
  Signals are not weights. Weights are not positions. Orders are not fills. Pick
  the mode that matches the research question before writing code.
</div>

## Choose The Right Mode

| If your idea says... | Use | Main hook | Read next |
|---|---|---|---|
| "Hold these assets at these portfolio weights" | weights | <code>_compute_weights()</code> | <a href="/engine/weights">Weights</a> |
| "Rank a universe, buy top names, rebalance monthly" | weights | <code>_compute_signals()</code> + <code>_compute_weights()</code> | <a href="/engine/rebalancing">Rebalancing</a> |
| "Keep target weights, but test lots, fills, margin and liquidity" | weights + orders | <code>_compute_weights()</code> + <code>weight_execution="orders"</code> | <a href="/engine/rebalancing">Rebalancing</a> |
| "Use inverse volatility, caps, cash sleeves or risk overlays" | weights | <code>_compute_weights()</code> | <a href="/engine/risk-models">Risk Models</a> |
| "Buy when a trigger fires and attach stops or limits" | orders | <code>_compute_orders(...)</code> | <a href="/engine/order-lifecycle">Order Lifecycle</a> |
| "Test market, limit, stop, trailing, bracket or OCO behavior" | orders | <code>_compute_orders(...)</code> | <a href="/engine/market-orders">Orders</a> |
| "I only need realized holdings from weights" | weights | no custom position hook | <a href="/engine/positions">Positions</a> |

Use weight mode first when the strategy is about allocation. Move to order mode
when execution details change the answer.

## Strategy Contract

Before coding, write a short contract. This catches most ambiguous strategy
requests before they become bugs.

```yaml
strategy_name: my_strategy_name
execution_mode: weights | orders
universe:
  symbols: [AAPL, MSFT, NVDA]
  source: yfinance | quantjourney_cloud
period:
  start: "2020-01-01"
  end: "2025-01-01"
benchmark:
  symbol: "^GSPC"
  name: "S&P 500 Index"
signal_timing: close_t
execution_timing: t_plus_1_weights | order_engine
exposure:
  long_only: true
  cash_allowed: true
  max_position_size: 0.25
rebalance:
  frequency: D | W | BME | BQE
  drift_threshold: null
costs:
  slippage: explicit_assumption
  commission: explicit_assumption
reports:
  text: true
  plots: true
  pdf: optional
```

## Weight Mode Skeleton

Use this shape for allocation research, ETF rotations, factor portfolios,
ranking models, inverse-vol portfolios and scheduled rebalancing.

<div className="qj-code-card">
<div className="qj-code-card-title">Minimal weight strategy</div>

```python
import asyncio
import os

import pandas as pd

from backtester import Backtester
from backtester.portfolio.rebalance import RebalancePolicy


class MyWeightStrategy(Backtester):
    def _compute_signals(self) -> pd.DataFrame:
        fast = self.instruments_data.get_feature("SMA_50_close")
        slow = self.instruments_data.get_feature("SMA_200_close")
        valid = fast.notna() & slow.notna()
        return (fast > slow).astype(float).where(valid, 0.0).fillna(0.0)

    def _compute_weights(self) -> pd.DataFrame:
        active = self.signals == 1.0
        counts = active.sum(axis=1)
        weights = active.div(counts, axis=0).fillna(0.0)
        return weights.clip(upper=0.25)


async def main() -> None:
    strategy = MyWeightStrategy(
        api_key=os.environ.get("QJ_API_KEY"),
        strategy_name="MyWeightStrategy",
        initial_capital=100_000,
        instruments=["AAPL", "MSFT", "NVDA", "GOOGL", "AMZN"],
        backtest_period={"start": "2020-01-01", "end": "2025-01-01"},
        source="yfinance",
        execution_mode="weights",
        max_position_size=0.25,
        rebalance_policy=RebalancePolicy(frequency="BME"),
        indicators_config=[
            {"function": "SMA", "price_cols": ["close"], "params": {"periods": [50, 200]}},
        ],
        benchmark_symbol="^GSPC",
        benchmark_name="S&P 500 Index",
        show_text_reports=True,
        save_text_reports=True,
        save_portfolio_plots=True,
    )
    await strategy.run_strategy()
    strategy.print_summary()


if __name__ == "__main__":
    asyncio.run(main())
```
</div>

### Weight Mode Checklist

- Return a dates x instruments DataFrame from <code>_compute_signals()</code>
  and <code>_compute_weights()</code>.
- Fill warmup NaNs explicitly.
- Keep row sums intentional: less than 1.0 means cash; more than 1.0 means
  leverage.
- Do not renormalize capped weights back to 100 percent if the strategy intends
  a cash sleeve.
- Use <code>RebalancePolicy(...)</code> to decide when targets become realized
  weights.
- Inspect turnover when changing rebalance frequency.

### Upgrade Weights To Executable Orders

No strategy-hook rewrite is required:

```python
from backtester.execution import FixedBpsCommission, FixedBpsSlippage
from backtester.portfolio.rebalance import RebalanceAt, RebalancePolicy

strategy = MyWeightStrategy(
    ...,
    execution_mode="weights",
    weight_execution="orders",
    rebalance_policy=RebalancePolicy(
        frequency="BME",
        rebalance_at=RebalanceAt.OPEN,
    ),
    slippage_model=FixedBpsSlippage(bps=2.0),
    commission_scheme=FixedBpsCommission(bps=1.0),
    max_volume_participation=0.10,
)
```

The unshifted target from t becomes a pending order after t and may fill from
t+1. Do not add another manual shift. Inspect decision, submission and fill
flags separately when validating the run.

## Order Mode Skeleton

Use this shape when the edge depends on discrete orders, stops, limits,
trailing exits, bracket exits, OCO logic, fill price, slippage or commissions.

<div className="qj-code-card">
<div className="qj-code-card-title">Minimal order strategy</div>

```python
import asyncio
import os

import pandas as pd

from backtester import Backtester
from backtester.execution.order_types import Order, OrderSide, OrderType


class MyOrderStrategy(Backtester):
    def __init__(self, **kwargs):
        super().__init__(**kwargs)
        self._prev_signal = {}

    def _compute_orders(self, date, bars, current_positions, nav) -> None:
        fast = self.instruments_data.get_feature("SMA_20_close")
        slow = self.instruments_data.get_feature("SMA_50_close")
        if date not in fast.index:
            return

        for inst in self.instruments:
            if inst not in fast.columns or inst not in slow.columns:
                continue
            if pd.isna(fast.loc[date, inst]) or pd.isna(slow.loc[date, inst]):
                continue

            signal = 1 if fast.loc[date, inst] > slow.loc[date, inst] else 0
            prev = self._prev_signal.get(inst, 0)
            pos = current_positions.get(inst, 0.0)

            if signal == 1 and prev == 0 and pos == 0:
                qty = int(nav * 0.15 / bars[inst].close)
                if qty > 0:
                    self.fill_engine.submit(Order(inst, OrderSide.BUY, qty, OrderType.MARKET))

            elif signal == 0 and prev == 1 and pos > 0:
                self.fill_engine.cancel_all(instrument=inst)
                self.fill_engine.submit(Order(inst, OrderSide.SELL, pos, OrderType.MARKET))

            self._prev_signal[inst] = signal


async def main() -> None:
    strategy = MyOrderStrategy(
        api_key=os.environ.get("QJ_API_KEY"),
        strategy_name="MyOrderStrategy",
        initial_capital=100_000,
        instruments=["AAPL", "MSFT", "NVDA", "GOOGL", "AMZN"],
        backtest_period={"start": "2020-01-01", "end": "2025-01-01"},
        source="yfinance",
        execution_mode="orders",
        indicators_config=[
            {"function": "SMA", "price_cols": ["close"], "params": {"periods": [20, 50]}},
        ],
        show_text_reports=True,
        save_text_reports=True,
        save_portfolio_plots=True,
    )
    await strategy.run_strategy()
    strategy.print_summary()


if __name__ == "__main__":
    asyncio.run(main())
```
</div>

### Order Mode Checklist

- Set <code>execution_mode="orders"</code>.
- Use <code>current_positions</code> so the strategy does not submit duplicate
  entries every bar.
- Treat orders emitted by <code>_compute_orders(...)</code> as pending for later
  bar processing.
- Cancel stale stop, limit, bracket or OCO orders before manually flattening a
  position.
- Add explicit slippage and commission assumptions for execution-sensitive
  tests.
- With daily OHLCV, do not claim exact intraday sequence when both stop and
  take-profit levels are touched.

## Universe And Data

The simplest strategy passes an explicit list of instruments:

```python
instruments=["SPY", "QQQ", "TLT", "GLD", "IEF"]
```

That is best for public examples and reproducible research. Larger production
workflows should define how the universe is built: membership date, liquidity,
asset class, survivorship handling, benchmark and exclusions. If a universe
comes from an API or saved screen, include the universe id or query in the
strategy contract.

## Rebalance Patterns

Weight-mode rebalancing is portfolio timing, not order simulation.

```python
from backtester.portfolio.rebalance import RebalancePolicy

# Daily.
RebalancePolicy(frequency="D")

# Weekly, Friday.
RebalancePolicy(frequency="W", weekday=4)

# Business month-end.
RebalancePolicy(frequency="BME")

# Month-end, or earlier when realized weights drift by 5 percent.
RebalancePolicy(frequency="BME", drift_threshold=0.05)

# Rebalance when signals change.
RebalancePolicy(frequency="BME", rebalance_on_signal_change=True)
```

For order-mode strategies, use order state and cancellations instead of
<code>RebalancePolicy</code>.

## AI Prompt Pack

Use these prompts with Codex, ChatGPT or another coding assistant. Keep the
contract visible in the prompt and require the assistant to choose the correct
execution mode before writing code.

### Prompt: Create A Strategy

````text
You are writing a QuantJourney Backtester strategy.

Use this strategy idea:

<STRATEGY_IDEA>
Describe the thesis, asset universe, timeframe, benchmark, indicators,
position sizing, rebalance rule, costs and risk constraints.
</STRATEGY_IDEA>

Return your answer in this order:

1. A short strategy contract in YAML.
2. The chosen execution mode: weights or orders.
3. The reason for that mode.
4. A complete runnable Python strategy file for the strategies/ directory.
5. The exact ./strategy.sh command to run it.
6. A short checklist of look-ahead, sizing, rebalance and reporting assumptions.

Rules:

- Use Backtester as the base class.
- Use execution_mode="weights" when the strategy returns target exposure.
- Use execution_mode="orders" when the strategy submits market, limit, stop,
  trailing, bracket or OCO orders.
- Use QuantJourney Cloud/yfinance data through the strategy constructor.
- Do not use future returns, shift(-1), next-day close or revised data in signals.
- Fill warmup NaNs explicitly.
- Keep cash and max_position_size semantics explicit.
- For weight mode, use RebalancePolicy when applicable.
- For order mode, use current_positions and cancel stale pending orders.
- Do not invent unavailable APIs. If an API is uncertain, say so before coding.
````

### Prompt: Create Weight Variants

````text
Create five QuantJourney weight-mode variants of this base strategy:

<BASE_STRATEGY>
Paste the current strategy file or describe the current thesis.
</BASE_STRATEGY>

For each variant:

- Change exactly one major dimension: universe, indicator, ranking rule,
  rebalance rule, risk cap or cash sleeve.
- Keep the strategy simple enough to run as a public example.
- Save each variant as a separate file under strategies/.
- Explain the hypothesis in the file docstring.
- Include the ./strategy.sh command for each variant.
- Avoid look-ahead bias and fill NaNs explicitly.
````

### Prompt: Create Order Variants

````text
Create QuantJourney order-mode strategies from this trading idea:

<TRADING_IDEA>
Describe entry, exit, sizing, stop/limit behavior, timeframe and universe.
</TRADING_IDEA>

Generate separate strategy files for these order types where they make sense:

- market
- limit
- stop
- stop-limit
- trailing stop
- trailing stop-limit
- bracket
- OCO

For every file:

- Use execution_mode="orders".
- Implement _compute_orders(date, bars, current_positions, nav).

- Cancel stale pending orders before manual exits.
- Add explicit slippage and commission assumptions.
- Explain daily OHLC assumptions in the docstring if stops or limits are used.
````

### Prompt: Review A Strategy

````text
Review this QuantJourney strategy as a senior quant engineer.

<STRATEGY_CODE>
Paste the full strategy file.
</STRATEGY_CODE>

Focus on:

- wrong execution mode
- look-ahead bias
- signal/weight/order/position confusion
- NaNs or misaligned DataFrames
- accidental leverage
- max_position_size and cash semantics
- rebalance policy mistakes
- stale pending orders
- unrealistic stop/limit assumptions on daily bars
- missing slippage, commission, benchmark or report settings

Return findings first, ordered by severity. Include file/line references when
possible. Then provide a minimal patch.
````

### Prompt: Debug A Failed Run

````text
Debug this QuantJourney strategy run.

Command:
<COMMAND>

Strategy file:
<STRATEGY_CODE>

Terminal output:
<OUTPUT>

Report metadata or traceback:
<METADATA_OR_TRACEBACK>

Find the smallest likely cause. Do not rewrite the strategy unless necessary.
Check credentials, universe, date range, indicators_config, execution_mode,
DataFrame shape, missing features, order types, pending orders and report paths.
Return a minimal fix and the command to rerun.
````
