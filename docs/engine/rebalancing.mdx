# Rebalancing

A weight strategy always writes target weights. The constructor decides how
those targets become portfolio state.

<div className="qj-contract">
  <div><strong>Policy</strong><code>RebalancePolicy</code></div>
  <div><strong>Fast path</strong><code>execution_mode="weights", weight_execution="fast"</code></div>
  <div><strong>Execution path</strong><code>execution_mode="weights", weight_execution="orders"</code></div>
  <div><strong>Outputs</strong>realized weights, quantities, NAV, costs, decision/submission/fill flags</div>
  <div><strong>Timing rule</strong>a target decided after bar t can fill no earlier than bar t+1</div>
</div>

## Choose The Accounting Contract

| Contract | What happens | Best for |
|---|---|---|
| <code>weight_execution="fast"</code> | Vectorized/loop weight accounting, implied trades and a weight cost model. | Ranking, factor, allocation and broad parameter research. |
| <code>weight_execution="orders"</code> | Stateful policy creates discrete contract-aware market orders, then FillEngine emits fills into PortfolioLedger. | Execution-aware portfolio research, FX/futures lots, margin, volume caps and audited rejects. |
| <code>execution_mode="orders"</code> | Strategy writes orders directly in <code>_compute_orders(...)</code>. | Stops, limits, brackets, OCO and custom order logic. |

The strategy's <code>_compute_weights()</code> implementation does not change
when moving from fast weights to execution-aware weights.

## Fast Weights

```python
strategy = MyStrategy(
    execution_mode="weights",
    weight_execution="fast",  # default
    rebalance_policy=RebalancePolicy(frequency="BME"),
    weight_cost_model=FixedBpsWeightCostModel(total_bps=2.0),
    ...,
)
```

The fast pipeline is:

```text
raw targets -> shift(1) -> cash buffer -> RebalanceEngine
            -> actual/end-of-bar weights + beginning-of-bar return weights
            -> implied trades and costs -> compatibility ledger
```

The first row is cash after the lag. A target calculated using close(t) is
applied at a later close and starts earning on the following return interval.

## Execution-Aware Weights

```python
from backtester.execution import FixedBpsCommission, FixedBpsSlippage
from backtester.portfolio.rebalance import RebalanceAt, RebalancePolicy

strategy = MyStrategy(
    execution_mode="weights",
    weight_execution="orders",
    rebalance_policy=RebalancePolicy(
        frequency="BME",
        rebalance_at=RebalanceAt.OPEN,
        drift_threshold=0.03,
        partial_rebalance=True,
        max_annual_turnover=6.0,
    ),
    slippage_model=FixedBpsSlippage(bps=2.0),
    commission_scheme=FixedBpsCommission(bps=1.0),
    max_volume_participation=0.10,
    ...,
)
```

The execution pipeline is:

```text
unshifted target[t] + realized ledger state after bar t
    -> ExecutionRebalanceEngine decision
    -> target quantity by ContractSpec
    -> reductions before increases
    -> conservative batch pre-trade checks
    -> pending orders
    -> next eligible bar pre-fill risk check
    -> fill -> PortfolioLedger -> realized exposure
```

Targets are deliberately not shifted before this controller. The order itself
creates the lag: it is submitted after bar t and cannot fill on bar t.

## Timing

| Setting | Earliest market fill for a close(t) decision |
|---|---|
| <code>RebalanceAt.OPEN</code> | open(t+1) |
| <code>RebalanceAt.CLOSE</code> | close(t+1) |
| <code>RebalanceAt.VWAP_WINDOW</code> | unsupported; constructor raises instead of using a silent proxy |

An explicit <code>fill_at="open"</code> or <code>fill_at="close"</code>
overrides OPEN/CLOSE timing by creating an equivalent policy copy. Same-bar
close fills are not permitted because the target may use that close.

## Stateful Reconciliation

The execution controller evaluates policy triggers against realized exposure,
not an idealized replay of target weights. This matters when an order is
partially filled, rejected, delayed by missing data or rounded to a valid lot.

- Unchanged target intent keeps its pending order instead of duplicating it.
- A changed target cancels stale generated orders and creates a new trade list.
- Planned sales do not finance purchases in another instrument before the
  sales actually fill.
- Rejected dependent legs are retried after realized reductions release
  buying power.
- Orders are never created on the final bar because no future fill is possible.
- Positive finite NAV is required for target sizing.

## Risk And Buying Power

Execution-aware weights default to
<code>PreTradeRisk(max_margin_utilization=1.0)</code>. Direct order strategies
retain the backward-compatible pass-through default unless a policy is
configured.

Risk is checked twice:

1. at submission using current marks and the complete trade list;
2. immediately before fill using the actual price after slippage and expected
   commission.

Every run publishes <code>margin_by_instrument</code>,
<code>margin_used</code>, <code>buying_power</code>,
<code>exposure_values</code> and <code>exposure_weights</code>.

## Contract-Aware Sizing

<code>ContractSpec</code> separates economic size from executable quantity:

- equity: integer shares by default;
- futures: integer contracts with multiplier and fixed/per-notional margin;
- FX: standard-lot notional with a default 0.01-lot quantity step;
- crypto: fractional quantity by default.

Rounding is toward zero, so unallocatable capital remains visible in cash.
<code>quantity_step</code> and <code>min_quantity</code> can be supplied by the
point-in-time instrument metadata returned by the data service.

## Policy Triggers

| Trigger or gate | Fields |
|---|---|
| Calendar | <code>frequency</code>, <code>weekday</code> |
| Absolute/relative drift | <code>drift_threshold</code>, <code>drift_type</code> |
| Tracking error | <code>tracking_error_threshold</code>, benchmark returns |
| Target change | <code>rebalance_on_signal_change</code> |
| Drawdown breaker | <code>max_drawdown_trigger</code>, action, cooldown |
| Turnover gate | <code>max_annual_turnover</code> |
| Partial rebalance | <code>partial_rebalance</code> |
| Tax heuristic | <code>avoid_short_term_gains</code> |
| Minimum order delta | <code>min_trade_size</code> in execution-aware weights |

Circuit-breaker exits bypass partial, tax, minimum-trade and turnover gates.
Cooldown starts only after the emergency target is reconciled; partial or dark
market exits remain pending. Post-cooldown re-entry is likewise kept alive
until its executable target is reconciled.

## Audit Fields

Execution-aware runs expose separate Boolean series:

- <code>rebalance_decision_flags</code>: the policy chose a target after this bar;
- <code>rebalance_submission_flags</code>: at least one generated order was accepted;
- <code>rebalance_fill_flags</code>: at least one generated order filled;
- <code>rebalance_flags</code>: planned execution dates for compatibility.

These are different by design. A decision may be rejected, and one trade list
may fill over several bars.

## Known Boundaries

- Daily OHLC cannot determine queue position or exact high/low sequence.
- Futures accounting currently uses a financing-adjusted fully-funded ledger;
  margin and exposure are reported, but this is not a broker variation-margin
  cash account.
- Quote-currency conversion, cash yield, borrow fees and funding are not yet
  modeled in the common ledger.
- Raw-price order accounting requires explicit split, dividend, roll and
  delisting events for production equity/futures research. Adjusted research
  prices are not a substitute for an auditable corporate-action ledger.
- The tax option is a holding-period heuristic, not a tax-lot optimizer.
- The turnover field currently measures gross traded notional/churn; report
  the convention rather than calling it one-way turnover.
